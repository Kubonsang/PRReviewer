v1.0 · Claude Code Skills

PRR

GitHub Pull Request에 AI 리뷰어가 코멘트를 남기는 도구. Claude Code 스킬로 동작하며, 유료 API 없이 ToS를 완전히 준수한다.

비용 0원 Claude Code 스킬 ToS 완전 준수 Private repo 지원
// 핵심 제약

설계 원칙

◈
비용 0원
유료 API 없음. Claude Code 구독 범위 내에서만 동작. 외부 서비스 의존 없음.
◉
ToS 완전 준수
사람이 직접 트리거하는 Claude Code 스킬 방식. 자동화 파이프라인이 없어 Consumer ToS §3-7 위반 없음.
◎
기능 최소화
설정 관리는 CLI, AI 작업은 스킬. 두 인터페이스의 역할을 명확히 분리.

시작하기

사전 요구사항

terminal
$claude --version  # Claude Code CLI
$gh auth status   # GitHub CLI 인증 확인
$python3 --version # Python 3.10+

PATH 등록

~/.zshrc
$echo 'export PATH="$PATH:/path/to/PRR"' >> ~/.zshrc
$source ~/.zshrc
$prr status  # 환경 확인

스킬 등록

terminal
$prr skill sync

prr skill sync는 PRR_DIR/skills/ 의 모든 스킬을 ~/.claude/skills/ 에 복사한다. git pull 이후나 스킬을 초기화할 때도 동일하게 사용한다.

전체 워크플로우

1
프로젝트 스캔 & 리포 등록
로컬 프로젝트를 스캔해 env.json을 생성한다. 설정 디렉터리와 기본 리뷰어도 함께 초기화된다.
/prr-scan ~/projects/my-repo owner/my-repo
2
프로젝트 스캔
Claude Code에서 스킬을 실행해 env.json을 자동 생성한다.
/prr-scan ~/projects/my-repo owner/my-repo
3
리뷰어 설정
AI가 프로젝트에 맞는 리뷰어 후보를 추천한다. 선택하면 페르소나를 자동 구성해 저장한다.
/prr-add-reviewer owner/my-repo
4
PR 리뷰 실행
PR 번호를 지정해 스킬을 실행하면 GitHub에 코멘트가 게시된다.
/prr-review owner/my-repo 42
// CLI 인터페이스

스킬 동기화

git pull로 PRR을 업데이트하거나, 커스텀한 스킬을 원본으로 되돌릴 때 사용한다. PRR_DIR/skills/ 의 스킬을 ~/.claude/skills/ 에 그대로 복사한다.

동기화

상태 확인
$prr skill status

// Claude Code 스킬

/prr-scan

로컬 프로젝트를 분석해 env.json을 자동 생성한다. 설정 디렉터리가 없으면 생성하고 기본 리뷰어도 함께 초기화한다. 별도의 prr init 없이 이 스킬 하나로 리포 등록이 완료된다.

/prr-add-reviewer

프로젝트 env.json을 읽어 기술 스택에 맞는 리뷰어 후보를 추천한다. 선택하면 페르소나·집중 항목·무시 항목을 자동으로 구성해 저장한다.

Claude Code 스킬
/prr-add-reviewer owner/my-repo
/prr-add-reviewer owner/repo │ ├─ env.json 로드 → 언어·프레임워크·의존성 파악 ├─ 기존 리뷰어 목록 확인 │ ├─ 리뷰어 후보 4–6개 추천 출력 (프로젝트 특성 기반) │ 예) 1. senior — 아키텍처·패턴 전반 │ 2. netcode — 네트워크 동기화·RPC 전문가 │ 3. performance — GC·풀링 최적화 전문가 │ ├─ 사용자 선택 ├─ persona / focus / ignore 자동 구성 → JSON 출력 ├─ 사용자 확인 [Y/n] └─ reviewers/{id}.json 저장
// Claude Code 스킬

/prr-review

등록된 모든 활성 리뷰어가 순서대로 PR을 분석한다. 발견된 이슈는 해당 코드 라인에 인라인 코멘트로, 나머지 요약(리뷰 기준·칭찬·마무리)은 별도의 PR 코멘트 하나로 게시한다.

Claude Code 스킬
/prr-review owner/my-repo 42
/prr-review owner/repo 42 │ ├─ env.json 로드 → exclude_paths, on_update 파악 ├─ gh pr diff 42 --repo owner/repo │ └─ exclude_paths 패턴 매칭으로 파일 블록 제거 │ ├─ on_update="review" 시 │ └─ 이전 PRR 코멘트 삭제 (<!-- PRR-AUTO-REVIEW --> 마커 기반) │ ├─ [junior.json] 리뷰어 ──────────────────────────────── │ ├─ 페르소나 채택 (focus / ignore / severity_threshold) │ ├─ diff 헝크에서 파일·라인 번호 계산 │ ├─ diff 검토 → 이슈 추출 (path + line + 설명) │ ├─ 인라인 코멘트 → 이슈를 각 코드 라인에 게시 │ └─ 요약 코멘트 → review_basis / praise / summary 게시 │ └─ [senior.json] 리뷰어 ──────────────────────────────── ├─ 페르소나 채택 ├─ diff 검토 → 이슈 추출 ├─ 인라인 코멘트 게시 └─ 요약 코멘트 게시
// Claude Code 스킬

/prr-followup

PR 작성자가 리뷰 코멘트를 반영한 커밋을 올린 뒤 실행한다. 이전 PRR 인라인 코멘트를 순회하며 수정이 확인된 항목에 답글을 자동으로 게시한다.

AI 코딩 툴 스킬
/prr-followup owner/my-repo 42
/prr-followup owner/repo 42 │ ├─ PR의 모든 review comment 수집 │ └─ <!-- PRR-INLINE --> 마커로 PRR 코멘트만 필터링 │ ├─ 이미 <!-- PRR-FOLLOWUP --> 답글이 달린 코멘트 제외 │ ├─ gh pr diff 로 현재 diff 수집 │ ├─ [코멘트별 수정 여부 판단] ─────────────────────── │ ├─ 지적 라인이 삭제·교체됐는가? │ ├─ 제안한 방식이 새 코드에 반영됐는가? │ └─ 불명확하면 건너뜀 (오탐 방지) │ ├─ 수정 확인된 코멘트마다 │ └─ ✅ 답글 게시 (<!-- PRR-FOLLOWUP --> 마커 포함) │ └─ 완료 메시지 (N건 중 M건 수정 확인)

답글 형식

GitHub 답글 예시
✅ 수정이 반영됐습니다! jwt.sign()에 expiresIn: '1h' 옵션이 추가됐습니다. 토큰 만료 처리가 올바르게 구현됐네요. <!-- PRR-FOLLOWUP -->

수정 여부가 불명확한 코멘트는 건너뛴다. 오탐(잘못된 수정 확인)이 미탐보다 해롭기 때문이다.

// 설정

env.json

리포 환경 정보와 리뷰 동작을 정의한다. /prr-scan 스킬이 자동 생성하며, 이후 수동으로 수정할 수 있다.

configs/repos/owner__repo/env.json
{ "repo": "owner/repo-name", "languages": ["TypeScript", "CSS"], "runtime": "Node.js 20", "framework": "Next.js 14", "test_framework": "Jest", "linter": "ESLint (airbnb-typescript)", "key_dependencies": ["prisma", "zod", "tailwindcss"], "conventions": [ "서버 컴포넌트 우선, use client 최소화", "함수형 컴포넌트만 사용" ], "review": { "exclude_paths": ["migrations/", "*.generated.ts", "*.lock"], "max_diff_lines": 500, "on_update": "skip" // "skip" | "review" } }
on_update동작
"skip"이전 PRR 코멘트를 유지 (기본값)
"review"이전 PRR 코멘트 삭제 후 새로 리뷰
// 설정

reviewer.json

리뷰어의 페르소나, 집중 항목, 무시 항목을 정의한다. /prr-review 스킬이 이 파일을 읽어 리뷰 방식과 코멘트 형식을 결정한다.

configs/repos/owner__repo/reviewers/junior.json
{ "id": "junior", "enabled": true, "comment_header": "🌱 Junior Reviewer", "persona": "당신은 1년차 주니어 개발자입니다...", "focus": [ "변수명·함수명이 역할을 명확히 드러내는가", "명백한 로직 오류나 누락된 예외처리", "불필요한 중복 코드" ], "ignore": [ "시스템 아키텍처 결정", "성능 최적화", "보안 취약점" ], "output_language": "ko", "severity_threshold": "low", "lgtm_comment": true, "tone": "친근하고 격려하는 말투. 동료 개발자를 응원하는 느낌으로.", "comment_sections": ["review_basis", "praise", "issues", "summary"] }

필드 레퍼런스

필드타입필수설명
idstring✓리뷰어 식별자. 파일명(id.json)과 일치해야 한다.
enabledboolean✓false로 설정 시 해당 리뷰어를 건너뛴다.
comment_headerstring✓GitHub 코멘트 상단에 표시되는 제목.
personastring✓리뷰어의 역할·관점·배경. 리뷰 방식 전체를 결정한다.
focusarray✓중점적으로 검토할 항목 목록.
ignorearray✓검토하지 않을 항목 목록.
output_languagestring—리뷰 코멘트 작성 언어. 예: "ko", "en". 없으면 persona 언어를 따른다.
severity_thresholdstring—보고할 최소 심각도. "low"(기본) · "medium" · "high". 이 값 미만의 이슈는 생략된다.
lgtm_commentboolean—이슈가 없을 때 LGTM 코멘트 게시 여부. 기본 true.
tonestring—코멘트 말투·태도. 없으면 중립적 서술체를 사용한다.
comment_sectionsarray—요약 PR 코멘트에 포함할 섹션 순서. 없으면 ["issues"].

comment_sections 옵션

값섹션 내용
"review_basis"집중 항목(focus)과 검토 제외 항목(ignore) 요약
"praise"diff에서 잘 작성된 코드·패턴 언급. 칭찬할 것이 없으면 섹션 생략
"issues"인라인 코멘트로 게시된 이슈 수 요약. 이슈 없으면 ✅ LGTM으로 대체
"summary"tone에 맞는 마무리 한마디

tone과 comment_sections는 선택 필드. 없으면 ["issues"]와 중립 말투를 사용한다.

예외 처리

1
Diff가 클 때
exclude_paths로 lock 파일, 생성 코드, 빌드 산출물을 사전 제거한다. Claude Code는 대용량 컨텍스트를 네이티브로 처리하므로 별도 분할 없이 전체 diff를 검토한다.
exclude_paths Claude 네이티브 처리
2
PR 재리뷰
on_update: "skip"이면 이전 코멘트를 그대로 유지 (기본값). "review"로 설정하면 <!-- PRR-AUTO-REVIEW --> 마커로 이전 PRR 코멘트를 식별·삭제하고 새로 리뷰한다.
skip review
3
LGTM (이상 없음)
리뷰어가 문제를 찾지 못하면 투명하게 LGTM 코멘트를 게시한다. lgtm_comment: false로 설정 시 코멘트 없이 조용히 통과.
LGTM ✓
4
리뷰 실패 시
gh CLI 오류나 설정 누락 시 스킬 내에서 즉시 에러 메시지로 피드백한다. macOS 데스크탑 알림(notify.sh)으로 추가 인지 가능.
에러 메시지 macOS 알림

실사용 예시

시나리오 — TypeScript 백엔드 리포 myorg/backend에 PR #42 "feat: JWT 인증 추가"가 올라왔다. 처음 등록하는 리포이고, 주니어·시니어 두 리뷰어를 붙이려 한다.

1단계 — 최초 세팅 (처음 한 번만)

1
리포 등록
설정 디렉터리와 기본 리뷰어 파일을 생성한다.
terminal
$prr init myorg/backend
✓ configs/repos/myorg__backend/ 생성됨
2
프로젝트 스캔
로컬 소스를 분석해 env.json을 자동 생성한다. AI 코딩 툴에서 실행한다.
AI 코딩 툴 스킬
/prr-scan ~/projects/backend myorg/backend
3
리뷰어 추가
AI가 프로젝트 특성에 맞는 리뷰어 후보를 추천한다. 선택하면 페르소나를 자동 구성해 저장한다.
AI 코딩 툴 스킬
/prr-add-reviewer myorg/backend
추천 리뷰어: 1) TS 타입 안전성 전문가 2) 보안 리뷰어 3) 주니어 ...
선택: 1, 2
✓ ts-safety.json, security.json 저장됨

2단계 — PR 리뷰 실행

4
PR 리뷰
PR 번호를 지정해 실행한다. 등록된 모든 활성 리뷰어가 순서대로 diff를 분석한다.
AI 코딩 툴 스킬
/prr-review myorg/backend 42
[ts-safety] diff 분석 중...
✓ 인라인 코멘트 2개 + 요약 코멘트 게시
[security] diff 분석 중...
✓ 인라인 코멘트 1개 + 요약 코멘트 게시
✓ PR #42 리뷰 완료 (2개 리뷰어)

3단계 — GitHub에서 결과 확인

각 이슈는 해당 코드 라인 아래 인라인 코멘트로 표시되고, 리뷰어별 요약은 PR 코멘트로 별도 게시된다.

인라인 코멘트 — src/auth/jwt.ts:38

🔐
Security Reviewer commented on src/auth/jwt.ts 인라인
36 const payload = { userId, role };
37 const secret = process.env.JWT_SECRET;
38+ const token = jwt.sign(payload, secret);
심각도: medium
jwt.sign()에 expiresIn 옵션이 없어서 토큰이 만료되지 않습니다. 탈취된 경우 무효화할 방법이 없어 보안 리스크가 생깁니다.

jwt.sign(payload, secret, { expiresIn: '1h' }) 형태로 만료 시간을 지정하는 걸 권장합니다.
🤖 이 리뷰는 AI가 작성했습니다.
